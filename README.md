# Тестовое задание - Analytics Engineer (Highcore)

## Коротко о задаче

Работа с сырыми событиями мобильной игры (Firebase).  
Цель - собрать витрины для анализа:

- retention новых пользователей
- монетизация по когортам

Стек: dbt + DuckDB

.venv\Scripts\activate

pip install -r requirements.txt

python scripts/prepare_data.py

dbt deps

dbt test

## Особенности данных

- данные событийные (1 строка = 1 событие)
- основной user id - `user_pseudo_id`
- параметры лежат в `event_params` (nested структура)

Чтобы с ними работать:
- использовал `UNNEST`
- приводил значения к одному типу

---

## Архитектура

Разбил модели на слои, чтобы не тащить лишнее и упростить логику.

### staging

**stg_events**
- распаковывает `event_params`
- приводит timestamp
- результат - “длинный” формат (event + param)

---

### intermediate

**int_events_pivot**
- переводит данные в “широкий” формат
- оставлены только нужные поля (board, screen и т.д.)

**int_users_first_seen**
- определяет дату первого события пользователя
- считается напрямую от raw (без pivot), чтобы не пересчитывать тяжелые модели

---

### marts

**mart_retention**
- считает retention по когортам
- использую факт активности пользователя в день (а не количество событий)

**mart_monetization**
- считает revenue и paying users
- используется поле `event_value_in_usd`

**mart_cohorts**
- финальная витрина
- объединяет retention и монетизацию
- пригодна для использования в BI без дополнительной логики

---

## Допущения

- user = `user_pseudo_id`
- retention считается по факту активности в день
- анализ ограничен первыми 30 днями после регистрации

### Монетизация

В данных очень мало событий с revenue.

Из-за этого:
- большинство значений = 0
- метрика присутствует, но сильно разрежена

В реальном проекте нужно было бы уточнить:
- какие события отвечают за покупки
- где находятся price / currency

## Тесты

Добавлены:

- not null
- уникальность (`cohort_date`, `day_number`)

Бизнес-проверки:

- retention в диапазоне [0, 1]
- retained_users ≤ cohort_users
- revenue ≥ 0
- day_number в диапазоне 0–30
