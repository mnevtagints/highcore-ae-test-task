# Тестовое задание: Analytics Engineer @ Highcore

Привет! Это тестовое задание на позицию Analytics Engineer.

Мы постарались его сделать близким к тому, как у нас выглядят реальные задачи. В нем используется публичный демо датасет игры от гугла "Flood-It!"

---

## TL;DR

Тебе нужно:

1. Разобраться в сыром потоке событий
2. Закрыть продуктовую задачу, спроектировать dbt-проект и витрины под неё
3. *(опционально) Оформить один шаг своей работы как переиспользуемый skill(Claude Code/Codex)*
4. Результат положить в Git-репозиторий

---

## Что мы хотим увидеть

В этом ТЗ мы смотрим на:

- **Hard skills** - dbt, SQL, проектирование моделей
- **Продуктовое мышление** - понимание продуктовых требований со стороны заказчика
- **Документацию**
- **Способность оформить процесс как переиспользуемый артефакт**

**Использование Claude Code / Codex / Cursor или любых других инструментов приветствуется**. Для нас навык умения работать над корнем проблемы стоит выше, чем чистый навык написания кода.

## Стек и сетап

Чтобы ты не тратил время на инфраструктуру, мы подготовили стартовый репозиторий с необходимым сетапом:

```
git clone https://github.com/gerasuchkov/highcore-ae-test-task
cd highcore-ae-test-task
make setup     # устанавливает зависимости и готовит данные
make build     # прогоняет dbt build
make docs      # поднимает dbt docs локально
```

**Что внутри:**

- **DuckDB** - как локальная замена BigQuery. SQL-диалект близок, включая работу с nested/struct-полями. В бою у нас BigQuery, для ТЗ - DuckDB, чтобы не было проблем с облачной авторизацией. Сфокусируйся на логике, не на инфраструктуре.
- **dbt-duckdb** - dbt-проект уже проинициализирован.
- **Данные** - события из мобильной F2P игры (публичный датасет firebase, выгружен в parquet и подгружен в DuckDB).

Ничего дополнительно ставить не потребуется

## Задание

### Легенда

Ты работаешь в роли Analytics Engineer в студии разработки мобильных игр. Есть поток игровых событий, который складывается в хранилище.

### Часть 1. Разведка данных

Ты получаешь доступ к сырым событиям игры. Тебе нужно разобраться, что там лежит.

**Что сделать:**

- Пойми структуру данных: какие события есть, как устроены nested-поля, какие есть пользовательские идентификаторы
- Найди аномалии в данных: дубли, пропуски, расхождения, необычные значения
- Сформулируй представление о том, как этот поток устроен, какие артефакты ты наблюдаешь (речь про структуру, не про продуктовые гипотезы)

**Артефакт:** `docs/data_exploration.md` - что нашёл, что показалось странным, аномалии в данных. Отчет в свободной форме.

### Часть 2. Продуктовая задача

Продакт пишет в чат:

> «Нам надо понять, как себя ведут новые игроки в первые дни. Хотим видеть retention, понимать монетизацию по когортам. Сделай нам простой дашборд, чтобы я мог сам туда заходить и смотреть.»

**Что сделать:**
- Разобраться, что на самом деле нужно продакту и правильно сформулировать задачу
- Принять решения по всем неоднозначностям и зафиксировать допущения с обоснованием.
- Реализовать ответ на задачу в виде dbt-моделей (Часть 3) и готовых витрин под дашборд (Часть 4)

**Подсказка:** нет "правильных" допущений, которые мы ожидаем со стороны кандидата. 

### Часть 3. dbt-проект

На основе сырых событий собери dbt-проект, который закрывает задачу из Части 2.

**Что ожидается:**

- Осмысленное разделение на слои (staging / intermediate / marts или твоя версия с обоснованием)
- Тесты - не только generic (`not_null`, `unique`), но и на бизнес-логику там, где это важно
- Документация моделей и ключевых полей (`dbt docs` должен давать осмысленную картину)
- Осознанный выбор materializations с объяснением, где это неочевидно
- `dbt build` корректно запускается

**Важно:** мы смотрим не на количество моделей, а на то, как ты думаешь об архитектуре. Не стоит тянуть все поля и параметры из сырой таблицы - бери то, что считаешь важным для задачи. Прокомментируй свой выбор.

### Часть 4. Витрины под дашборд и скетч

Подготовь необходимые витрины и мокап дэшборда

**Что надо сделать:**

- В marts-слое подготовь витрины, из которых дашборд собирается тривиально - то есть без дополнительной или с минимальной логикой на стороне BI
- Скетч дашборда: что на нём, какие разрезы, какие фильтры, как это отвечает на вопрос продакта. Скетч в любом формате: markdown-таблица, Figma, Miro, Excalidraw, что угодно. Главное - понятно донести идею. 
- Объясни, почему именно такие разрезы и фильтры

### Бонус. Кастомный скилл

Проанализируй, что ты делал в Частях 1-4. Выбери один шаг, который ты бы хотел автоматизировать, и оформи его как переиспользуемый skill.

Примеры (не ограничивайся ими):

- Скилл для разведки нового источника данных
- Скилл для генерации staging-моделей из схемы таблицы
- Скилл для написания dbt-тестов на бизнес-логику
- Скилл для проверки сходимости двух источников данных
- Скилл для генерации документации по dbt-модели

**Что ожидается:**

- Папка `skills/<название-скилла>/` с `SKILL.md` и всем необходимым
- Скилл работает
- Выбор задачи на автоматизацию не важен

### Часть 5. README и письменная часть

README - это то, по чему мы будем входить в твой проект. Сделай его так, чтобы по нему могли войти и разобраться с нуля.

**Рекомендуемые разделы:**

1. **Как запустить** - воспроизводимо с нуля
2. **Что я сделал и почему** - короткая карта решений
3. **Допущения и уточнения** - какие допущения зафиксировал и почему
4. **Trade-offs** - что выбрал, что бы сделал иначе при большем времени
5. **Как бы я оркестрировал это в Airflow** - текстом, без кода: какие DAG'и, какие зависимости, какое расписание, как обрабатывать сбои
6. **Вопросы, которые я задал бы в реальном проекте** - даже если на ТЗ их не задавал
